## 1. What We Reproduced

We fully reconstructed the four components of the data synthesis pipeline from the paper:

### 1.1 Hierarchical Scenario Tree (HST)
**Paper Section:** 4.1.1
**Implementation:** `src/hst.py`

The HST represents optimization problems as structured trees:
```
Problem Root
├── Scenario Node (e.g., "Logistics")
│   ├── Entity Subtree
│   │   ├── Variable: x (continuous) — "shipment quantity"
│   │   └── Variable: y (binary) — "route selected"
│   └── Constraint Subtree
│       ├── Constraint: capacity constraint
│       └── Constraint: demand constraint
└── Objective Function Node
    └── minimize: sum(cost * x)
```

**Verification:** ✅ Successfully instantiated HST for 10 sample problems. Tree structure, entity/constraint mapping, and semantic similarity scores ($S_{sem}$) computed correctly.

---

### 1.2 Scenario Union Operator
**Paper Section:** 4.1.2, Equation 4
**Implementation:** `src/evolution.py` → `scenario_union()`

**How it works:**
- Takes two problems $P_1$, $P_2$ where $S_{sem}(P_1, P_2) \geq \theta_{sem}$
- Merges their entity sets and constraint sets
- Creates a combined, more complex problem

**Example generated by our pipeline:**
```
Input 1: "Standard Logistics Optimization" (minimize shipping cost)
Input 2: "Cold-Chain Logistics" (temperature constraints)
Output:  "Cold-Chain Logistics with Multi-Depot" 
         (minimize cost + temperature constraints + depot capacity)
```

**Verification:** ✅ Union operator successfully merges compatible problem trees. Output problems have higher constraint count and more variables than inputs.

---

### 1.3 Scenario Transfer Operator
**Paper Section:** 4.1.2, Equation 5
**Implementation:** `src/evolution.py` → `scenario_transfer()`

**How it works:**
- Takes a problem $P_1$ (source domain) and $P_2$ (target domain)
- Where $S_{func}(P_1, P_2) \geq \epsilon$ (functionally similar math structure)
- Transfers the mathematical structure of $P_1$ to the domain context of $P_2$

**Example:**
```
Source: Portfolio optimization (finance domain)
Target: Resource allocation (manufacturing domain)
Output: "Manufacturing Resource Portfolio" 
        (same quadratic programming structure, new domain context)
```

**Verification:** ✅ Transfer operator correctly identifies functionally equivalent mathematical structures across domains.

---

### 1.4 Knowledge Fogging Operator
**Paper Section:** 4.1.2, Equation 6
**Implementation:** `src/evolution.py` → `knowledge_fogging()`

**How it works:**
- Intentionally **removes explicit domain-specific knowledge** from problem descriptions
- Forces the agent to retrieve missing knowledge from Arxiv during inference
- Creates training examples where search is required to solve the problem

**Example:**
```
Before Fogging: "Use the standard Branch-and-Bound algorithm to solve this MIP..."
After Fogging:  "Find the optimal solution for this integer programming problem..."
                (agent must look up solution methods via Arxiv search)
```

**Verification:** ✅ Fogging operator successfully strips domain knowledge markers. Agent correctly triggers `<search>` actions in response to fogged problems.

## 3. Quality Assessment

### Why HST synthesis is better than naive augmentation

| Approach | Diversity | Mathematical Validity | Requires Search | Domain Coverage |

| Naive paraphrasing | Low | High | No | Low |
| Template filling | Medium | High | No | Medium |
| **HST Synthesis (paper)** | **High** | **High** | **Yes** | **High** |

**Key insight:** Knowledge Fogging is what forces the agent to generalize — by training on problems where domain knowledge is absent, the model learns to search rather than memorize.
